# Техническое описание системы RestaurantFlow

## 1. Архитектура и общая концепция
**RestaurantFlow** — это микросервисное приложение для сквозного управления процессами в ресторане: от просмотра меню до оплаты заказа. 

* **Архитектурный подход:** Разделение бизнес-логики на независимые слабосвязанные сервисы с единой точкой входа (API Gateway).
* **Взаимодействие:** Событийно-ориентированная архитектура (Event-Driven) для обмена статусами между сервисами.

```
                  [ Клиентский интерфейс (Web / Mobile) ]
                                     │
                                     ▼
                            [ API Gateway ]
                                     │
      ┌──────────────────────┬───────┴──────────────┬──────────────────────┐
      ▼                      ▼                      ▼                      ▼
[ Auth Service ]     [ Menu Service ]       [ Order Service ]     [ Payment Service ]
(Сессии / JWT)       (Каталог / CRUD)       (Статусы / Жизн. цикл) (Транзакции / Возвраты)
```

---

## 2. Матрица ролей и прав доступа

| Роль | Функциональные возможности |
| :--- | :--- |
| **Пользователь** | Регистрация/авторизация, просмотр категорий меню, создание, оплата и отмена заказов, получение push-уведомлений о статусах. |
| **Администратор** | Управление каталогом меню (CRUD), редактирование цен, контроль и принудительная отмена заказов (при отсутствии позиций). |

---

## 3. Спецификация микросервисов

### 3.1 Auth Service (Сервис авторизации)
Отвечает за аутентификацию, разграничение прав доступа и ведение профилей.
* **Вход в систему:** Беспарольный доступ по номеру телефона со случайным SMS-кодом. Выдача токена доступа.
* **Профиль:** Изменение контактных данных пользователем; аудит и просмотр карточек клиентов администратором.
* **Сессии:** Инвалидация и аннулирование токенов при выходе (Logout).

### 3.2 Menu Service (Сервис меню)
Выступает в роли хранилища и витрины доступных позиций.
* **Для пользователей:** Чтение и фильтрация каталога по структурированным категориям (*Пицца, Суши, Напитки*).
* **Для администратора (CRUD):** Создание позиций, обновление описания/цен, переключение статуса доступности (в наличии / стоп-лист), удаление.

### 3.3 Order Service (Сервис заказов)
Управляет жизненным циклом и бизнес-логикой обработки заказов.
* **Оформление:** Сбор выбранных позиций, валидация состава, присвоение `UUID` заказа.
* **Жизненный цикл статусов:**
    ```
    [Создан] ──> [Готовится] ──> [Готов] ──> [Выдан]
       │             │
       └─────┬───────┘
             ▼
         [Отменён] (Клиентом или Администратором)
    ```
* **Триггеры изменений:** Внешние события (успешная оплата), ручные действия персонала или системные таймеры. По каждому шагу отправляется push/SMS уведомление клиенту.

### 3.4 Payment Service (Сервис оплаты)
Шлюз для проведения финансовых транзакций и интеграции с эквайрингом.
* **Проведение платежа:** Инициация транзакции, обработка реквизитов, фиксация исхода (`Успешно` / `Ошибка`).
* **Синхронизация:** Отправка асинхронного события «Оплата прошла» в *Order Service* для автоматического изменения статуса заказа.
* **Чарджбэк (Refund):** Оформление и проведение возвратов средств на счет клиента при отмене заказа.
